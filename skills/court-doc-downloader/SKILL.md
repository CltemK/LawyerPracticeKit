# court-doc-downloader

## Description

法院送达文书下载工具（v7）。从 zxfw.court.gov.cn 下载所有送达 PDF，自动进行文字提取与 OCR 识别，根据文书类型自动命名，上传至飞书云空间（按案号归类），并为传票创建飞书团队日历开庭日程。

## Use Cases

用户发送包含法院送达链接的消息时使用。完整流程：获取文件列表 → 按案号分组 → 逐个下载到临时目录 → PDF 文字提取 → 文书识别 → 自动命名 → 上传飞书云空间 →（传票则创建飞书日历）。

## Execution

```bash
# 默认上传到飞书云空间根目录，按案号自动归类
python court_doc_downloader.py <URL>

# 指定飞书父文件夹（传入 folder_token）
python court_doc_downloader.py <URL> --parent-folder <folder_token>

# 跳过飞书日历创建
python court_doc_downloader.py <URL> --skip-calendar

# 只下载指定文件
python court_doc_downloader.py <URL> --files "1,3,5"
```

## Output

- **不再保存到本地**，下载至系统临时目录后直接上传至飞书云空间，上传成功后自动清理本地临时文件，不占用本地空间
- **飞书文件夹命名规则**：`{案号}_{最新送达日期}`（如 `（2026）鄂0103民初1234号_20260411`）
  - 案号：从文书名称（c_wsmc）中提取；无法提取时标记为"未知案号"
  - 最新送达日期：该案号下所有文件中最早的 `dt_cjsj`（送达文件创建时间），格式 YYYYMMDD
  - 同一案号的多份文件（如传票+判决书+通知书）全部归入同一文件夹
- **同名文件自动加序号避免覆盖**
- **传票**重命名: `开庭日期_法院名称_传票.pdf`（如 `2026年04月16日14:50_武汉市江汉区人民法院_传票.pdf`）
- **判决书**重命名: `原告与被告+案由+纠纷_读取日期.pdf`
- **其他文书**（须知/通知/告知书等）：使用 API 返回的文书名称，含案号的文书加案号后缀区分
- **传票**自动在飞书**团队日历**中创建开庭日程

## 处理流程

1. **获取文件列表**：调用法院 API `getWsListBySdbhNew` 获取所有送达文书（支持多文件）
2. **按案号分组**：
   - 从每份文书的名称（c_wsmc）中提取案号
   - 无法提取时该文件独立成组（以文书名称作为标识）
   - 记录每个案号组中最新的 `dt_cjsj`（送达时间）
3. **创建飞书文件夹**：
   - 检查目标飞书文件夹中是否已有同名文件夹（有则复用）
   - 无则创建，文件夹名称为 `{案号}_{最新送达日期}`
   - 支持指定 `--parent-folder` 上传到指定文件夹，否则上传到云空间根目录
4. **逐个下载**：通过 PDF 直链（含签名）下载到临时目录，同名文件自动加序号避免覆盖
5. **文字提取**：
   - 优先用 `pdfplumber` 提取文本（文本型 PDF）
   - 文字不足时调用 `mineru-extract` skill 的 `mineru_parse_documents.py` 对 PDF 直链 URL 做 OCR（扫描型 PDF）
6. **文书识别**：根据内容关键词判断类型（传票 / 判决书 / 其他）
7. **自动命名**：
   - 传票：提取"开庭日期"+"法院名称" → `2026年04月16日14:50_武汉市江汉区人民法院_传票.pdf`
   - 判决书：提取"我所代理的原告"+"被告"+"案由" → `张三与李四民间借贷纠纷_2026年01月15日.pdf`
   - 其他：使用原始文书名称，有案号的加案号后缀
8. **上传飞书**：重命名后立即上传至对应案号的飞书文件夹，上传成功后清理本地临时文件
9. **飞书团队日历**（仅传票）：调用 `lark-cli calendar events create` 创建开庭日程

## 文书类型识别规则

- 标题/内容含"传票" → **传票**
- 含"判决书"字样 → **判决书**
- 含"须知/通知/指南/告知书/权利义务" → **其他**
- 原告/被告各出现 >=3 次 → **判决书**（高置信度）
- 其他情况按关键词评分判断

## 判决书解析规则（重要）

### 文书处理原则
- **撤诉裁定书**：不需要解析，内容无实质性
- **判决书**：需要解析，重点在"本院认为"部分

### 解析目标
解析判决书时，提取以下结构化信息：
1. **争议焦点**：当事人核心争议是什么
2. **裁判观点**：法院对本案的实质性判断（重点）
3. **裁判结果**：判决结论（支持/驳回/部分支持）

### 一审判决书解析方法
- 本院认为**只出现一次**
- "本院认为"后面的内容就是裁判观点
- 格式：`本院认为，...` → 提取冒号或逗号后的整段

### 二审判决书解析方法
- 本院认为**通常出现两次**
  - "一审法院认为"：一审法院的观点（可忽略）
  - "本院认为"：二审判决法院的观点（**重点**）
- **只需总结二审法院的"本院认为"部分**
- 格式：`本院认为，...` → 提取冒号或逗号后的整段

### 解析示例
```
一审："...本院认为，被告未按约定履行合同义务..."
  → 裁判观点：被告未按约定履行合同义务

二审："...一审法院认为，原告主张的损失与被告无因果关系..."
  → 忽略一审观点
  "...本院认为，原告提交的证据足以证明损失存在..."
  → 裁判观点：原告提交的证据足以证明损失存在
```

### 注意事项
- 撤诉裁定书直接跳过，不做实质性解析
- 二审优先看"本院认为"，忽略"一审法院认为"
- 裁判观点的核心在本院认为之后的内容

## 环境变量

| 变量 | 用途 | 默认值 |
|------|------|--------|
| `COURT_FEISHU_FOLDER_TOKEN` | 飞书上传目标父文件夹 token | 空（根目录）|
| `COURT_TEAM_CALENDAR_ID` | 飞书团队日历 ID（用于创建开庭日程） | 空（必填）|
| `COURT_OUTPUT_DIR` | 上传失败时本地备份目录 | `./court_documents` |
| `LARK_CLI_PATH` | lark-cli 可执行文件路径（可选） | 自动查找 |
| `MINERU_EXTRACT_SCRIPT` | MinerU OCR 脚本路径（可选） | `mineru_parse_documents.py`（同目录）|

## 依赖

| 依赖 | 用途 | 状态 |
|------|------|------|
| `pdfplumber` | 文本型 PDF 文字提取 | 已预装 |
| `mineru-extract` skill | 扫描型 PDF OCR（MinerU API） | 已安装（通过 skill 本地 .env 配置 MINERU_TOKEN） |
| `lark-cli` | 飞书云空间上传 & 团队日历创建 | 需已安装并授权 |
